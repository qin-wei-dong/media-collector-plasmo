## 本次遇到的问题

### 1. `SUBMIT_KEYS` 没有在 Actions 中生效

现象：

- 早期发布流程依赖 `SUBMIT_KEYS`
- 仓库 Actions 里没有可用的发布凭据
- `PlasmoHQ/bpp@v3` 需要可用的 Browser Platform Publish JSON key

处理：

- 将 `BPP_KEYS` 作为正式 secret
- `SUBMIT_KEYS` 只保留为历史兼容 fallback
- 更新发布文档，明确以后统一配置 `BPP_KEYS`

### 2. 之前没有配置过 JSON 凭据

现象：

- 本地没有可用于 `BPP_KEYS` 的 Chrome Web Store OAuth JSON
- 需要从 Google OAuth client 获取 `refreshToken`

处理：

- 使用用户已创建的 Google OAuth Desktop app
- 通过 OAuth 授权获取 code
- 换取 `refreshToken`
- 生成 `BPP_KEYS` 所需 JSON
- 使用 `gh secret set BPP_KEYS --repo qin-wei-dong/media-collector-plasmo < key.json` 写入仓库 secret

注意：

- `key.json` 只作为临时文件使用
- 不提交到仓库
- 不在聊天或日志里输出 secret 值

### 3. Google OAuth 测试用户缺失

现象：

- OAuth consent screen 处于 Testing
- 授权账号未加入 test users 时会被 Google 拦截

处理：

- 将发布账号加入 Google Auth Platform 的 test users
- 重新走 OAuth 授权流程

### 4. token 换取阶段网络超时

现象：

- 本地工具已拿到 OAuth callback code
- 换取 token 时请求 `https://oauth2.googleapis.com/token` 超时
- shell 没有走到浏览器可用的网络路径

处理：

- 读取系统代理配置
- 通过本地代理重新请求 token endpoint
- 成功换取 `refreshToken`

经验：

- 如果 OAuth callback 已拿到 `code=...`，不要先怀疑 OAuth 配置
- token endpoint 超时优先排查 shell 网络出口 / 代理

### 5. `PlasmoHQ/bpp` 返回 403

失败 run：

- <https://github.com/qin-wei-dong/media-collector-plasmo/actions/runs/27969780747>

现象：

- `Check publish keys` 已通过
- `Browser Platform Publish` 失败
- 日志中出现 `Response code 403 (Forbidden)`

真实原因：

- Google Cloud 项目未启用 `Chrome Web Store API`
- API 返回 `accessNotConfigured`

处理：

- 打开 Google Cloud 的 `Chrome Web Store API` 页面
- 启用 `chromewebstore.googleapis.com`
- 等待权限传播
- 重新触发 `Submit to Web Store`
- 新 run 成功通过

## 后续注意事项

1. 每次发版前确认 `package.json` 版本号已更新。
2. 每次发版前确认 `CHANGELOG.md` 已补本次版本记录。
3. 发布只走 GitHub Actions `Submit to Web Store`，不要绕过工作流手动上传。
4. `BPP_KEYS` 是正式发布 secret，`SUBMIT_KEYS` 只作为历史兼容。
5. 不要提交或外发 `key.json`、`clientSecret`、`refreshToken`。
6. 如果发布时报 403，优先检查 Google Cloud `Chrome Web Store API` 是否启用。
7. 如果 OAuth 被拦，优先检查 test users 和当前登录账号。
8. 如果 token endpoint 超时，优先检查 shell 网络代理。
9. 发布成功后继续观察 24-48 小时，关注商店邮件、用户反馈和核心流程异常。
